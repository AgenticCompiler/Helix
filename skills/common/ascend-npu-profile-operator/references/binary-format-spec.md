## MindStudio Profiler binary output format specification

This document describes the binary file format produced by the MindStudio Profiler (msprof) when profiling operators on Huawei Ascend NPU hardware. The profiler emits binary files containing embedded JSON data blocks that describe operator execution characteristics, pipe utilization, workload breakdown, memory bandwidth, and optimization advice.

### 1. Binary file structure

The binary output file contains **5 sequential JSON blocks**. Each block is preceded by a `ZZ{` marker in the binary stream. To extract the JSON data:

1. Scan the binary content for all occurrences of the byte sequence `ZZ{`.
2. Starting from the `{` character (skipping the `ZZ` prefix), parse a complete JSON object.
3. The 5 blocks appear in a fixed order (Block 0 through Block 4), each containing different profiling data as described below.

### 2. Block 0 -- Base info

Contains top-level operator execution metadata.

| Field | Type | Description |
|-------|------|-------------|
| `name` | string | Operator name as executed on the NPU |
| `duration` | number | Execution time of the operator (in microseconds) |
| `op_type` | string | One of `"vector"`, `"cube"`, or `"mix"` |
| `block_dim` | integer | Number of AI cores used |

The block also contains execution detail in one of two formats depending on `op_type`:

- **vector or cube operators**: a `block_detail` object with `head_name` (array of column names) and `rows` (array of row arrays).
- **mix operators**: a `mix_block_detail` object with the same `head_name` / `rows` structure, combining both vector and cube execution data.

### 3. Block 1 -- Pipe utilization

Reports utilization percentage for each execution pipeline.

| Field | Type | Description |
|-------|------|-------------|
| `subblock_detail` | array | One entry per pipeline stage |

Each entry in `subblock_detail`:

| Field | Type | Description |
|-------|------|-------------|
| `block_id` | integer | Identifies which AI core or block |
| `name` | string | Pipeline name (e.g. `"Cube"`, `"Vector"`, `"Scalar"`, `"MTE1"`, `"MTE2"`, `"MTE3"`) |
| `value` | number | Utilization percentage (0--100) |

### 4. Block 2 -- Workload details

Provides per-block instruction-level breakdown.

| Field | Type | Description |
|-------|------|-------------|
| `subblock_detail` | array | One entry per instruction or special metric |

Each entry in `subblock_detail` has `block_id`, `name`, and associated values. There are three categories of entries:

| Category | `name` pattern | Fields present | Description |
|----------|---------------|----------------|-------------|
| Wait | `"Vector Wait"` | `duration` only | Time spent waiting (no instruction count) |
| Data volume | `"*Compute Data Size"` | `data_volume` only | Total data moved (no instruction count) |
| Normal instruction | Any other string | `instruction_name`, `instruction_count` | Instruction name and how many times it executed |

### 5. Block 3 -- Core memory map

Per-core memory bandwidth and cache statistics.

| Field | Type | Description |
|-------|------|-------------|
| `core_memory_map` | array | One entry per AI core |

Each entry is keyed by `core_no` (integer) and contains:

#### `memory_unit` array

Each element describes one data path:

| Field | Type | Description |
|-------|------|-------------|
| `memory_path` | integer | Data path ID (0--19, see section 7) |
| `bandwidth` | number | Measured bandwidth in GB/s |
| `request_count` | integer | Number of memory requests on this path |

#### Optional cache/unit statistics

These fields appear when applicable to the core's workload:

| Field | Type | Description |
|-------|------|-------------|
| `L2cache` | object | Contains `hit_ratio` (0.0--1.0) |
| `Vector` | object | Contains `ratio` (utilization of Vector0 unit) |
| `Vector1` | object | Contains `ratio` (utilization of Vector1 unit) |
| `Cube` | object | Contains `ratio` (utilization of Cube unit) |

#### `advice` array

Array of strings containing hardware-specific optimization hints generated by the profiler (e.g. suggestions to improve data locality or reduce memory stalls).

### 6. Block 4 -- Memory workload tables

Detailed per-block memory workload data with optimization advice.

| Field | Type | Description |
|-------|------|-------------|
| `table_per_block` | array | One entry per block |

Each entry is keyed by `block_id` (integer) and contains:

| Field | Type | Description |
|-------|------|-------------|
| `advice` | array | Optimization hint strings specific to this block |
| `table_detail` | object | Contains `header_name` (array of column names) and `rows` (array of row arrays) |

### 7. Ascend NPU data path reference

The `memory_path` field in Block 3 maps to the following 20 data paths in the Ascend NPU memory hierarchy:

| Path ID | Source | Destination | Notes |
|---------|--------|-------------|-------|
| 0 | GM (Global Memory) | L2 Cache | DDR to shared cache |
| 1 | L2 Cache | GM (Global Memory) | Shared cache writeback to DDR |
| 2 | L2 Cache | L1 | Shared cache to per-core data cache |
| 3 | L1 | L2 Cache | Per-core data cache writeback |
| 4 | L1 | L0A | Matrix input A buffer load |
| 5 | L1 | L0B | Matrix input B buffer load |
| 6 | L0A | Cube | Feed matrix A to Cube unit |
| 7 | L0B | Cube | Feed matrix B to Cube unit |
| 8 | Cube | L0C | Cube unit output to result buffer |
| 9 | L0C | Cube | Accumulation feedback path |
| 10 | L0C | L2 Cache | Result buffer writeback to shared cache |
| 11 | L0C | L1 | Result buffer to per-core data cache |
| 12 | L2 Cache | UB | Vector0: shared cache to unified buffer |
| 13 | UB | L2 Cache | Vector0: unified buffer writeback |
| 14 | UB | Vector | Vector0: unified buffer to compute unit |
| 15 | Vector | UB | Vector0: compute result to unified buffer |
| 16 | L2 Cache | UB | Vector1: shared cache to unified buffer |
| 17 | UB | L2 Cache | Vector1: unified buffer writeback |
| 18 | UB | Vector | Vector1: unified buffer to compute unit |
| 19 | Vector | UB | Vector1: compute result to unified buffer |

### 8. Memory hierarchy diagram

```
GM (Global Memory / DDR)
  |
  v
L2 Cache (Shared across cores)
  |
  +---> L1 Data Cache (Per-core)
  |       |
  |       +---> L0A (Matrix input A) ---> Cube Unit
  |       |                                   |
  |       +---> L0B (Matrix input B) ---> Cube Unit
  |       |                                   |
  |       +---> L0C (Output buffer)  <--- Cube Unit
  |       |       |
  |       |       +---> L0C feedback ---> Cube Unit (accumulation)
  |       |
  |       +---> UB (Unified Buffer) <---> Vector Unit (Vector0)
  |       |
  |       +---> UB (Unified Buffer) <---> Vector Unit (Vector1)
  |
  +---> UB (Unified Buffer, via L2 directly for Vector paths 12-19)
```

Data flow summary:
- **Cube path**: GM -> L2 -> L1 -> L0A/L0B -> Cube -> L0C -> L1/L2 -> GM
- **Vector path**: GM -> L2 -> UB -> Vector -> UB -> L2 -> GM

### 9. Interpretation guide

#### Operator type (`op_type`)

| Value | Meaning | Typical kernels |
|-------|---------|-----------------|
| `vector` | Element-wise or reduction operations using Vector units | Activation functions, normalization, element-wise arithmetic |
| `cube` | Matrix multiplication operations using the Cube unit | GEMM, linear layers, batched matmul |
| `mix` | Combined vector and cube operations in a single kernel | Fused attention, fused MLP layers |

#### Key performance indicators

| Metric | Location | Healthy threshold | Issue if violated |
|--------|----------|-------------------|-------------------|
| Pipe utilization | Block 1 `value` | > 60% | Low utilization means the pipeline is stalling or underused; look for data dependency stalls or insufficient parallelism |
| L2 cache hit ratio | Block 3 `L2cache.hit_ratio` | > 0.80 (80%) | Below 80% indicates poor data locality; consider tiling, data layout changes, or prefetching |
| Memory bandwidth | Block 3 `bandwidth` | Close to theoretical peak | Bandwidth significantly below theoretical maximum means the memory bus is underutilized; check for strided access patterns or small transfer sizes |
| Vector Wait duration | Block 2 special entry | Minimal relative to total | High wait time indicates the Vector unit is blocked waiting for data; memory-bound bottleneck |

#### Common optimization patterns

- **Low pipe utilization + high wait time**: The compute unit is starved for data. Optimize memory access patterns or increase data prefetching.
- **Low L2 hit ratio**: Working set exceeds cache capacity or access pattern is not cache-friendly. Apply tiling to fit working data in L2.
- **High bandwidth on GM->L2 but low on L2->L1**: Data is being fetched from global memory repeatedly instead of being reused from cache. Improve temporal locality.
- **Cube utilization low but Vector utilization high**: The kernel is spending most time on element-wise work rather than matrix computation. Consider fusing element-wise operations or restructuring to maximize Cube unit usage.
- **Unbalanced Vector0 vs Vector1 utilization**: Work is not evenly distributed across the two vector units. Rebalance workload or check for serialization between vector operations.
