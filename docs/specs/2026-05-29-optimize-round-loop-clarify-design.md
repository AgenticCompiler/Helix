# Optimize Round Loop Clarify Design

## Summary

- Remove the runtime-specific `pass-stop` and `pass-continue` meanings from supervisor decisions.
- Treat `check-round` as the technical acceptance authority for round validity.
- Let the CLI loop decide whether another worker round is required based on `min_rounds`.
- Keep `supervised` mode as `worker -> technical validation -> supervisor audit`, but narrow supervisor so it no longer decides stop versus continue.

## Problem

The current `run_round_loop` mixes two different concerns into one `GateDecision` enum:

- whether a round is technically acceptable
- whether the outer loop should keep running

That makes the loop hard to follow, and it gives `supervised` mode an odd control flow where supervisor can override the `min_rounds` requirement through `pass-stop`.

## Goals

- Make `checked` and `supervised` loops read as explicit state transitions instead of intertwined branching.
- Keep technical validation and orchestration decisions separate.
- Keep supervisor focused on audit and metadata repair, not session termination.

## Non-Goals

- Do not redesign the round artifact contract in this change.
- Do not remove `round_disposition` from round-state artifacts in this change.
- Do not change `check-round` result kinds away from `pass`, `revise-required`, and `hard-fail`.

## Design

### Technical Validation

`check-round` remains the authority for round acceptance:

- `pass`
- `revise-required`
- `hard-fail`

The runtime should not reinterpret a passing round as `pass-stop` or `pass-continue`.

Instead, the runtime should carry loop continuation separately as a boolean derived from:

- current completed round count
- requested `min_rounds`

### Checked Loop

`checked` mode should run:

1. worker round
2. technical validation
3. if validation fails, repair or fail
4. if validation passes and `min_rounds` is not yet satisfied, continue with an injected CLI summary
5. otherwise stop and return the worker result

### Supervised Loop

`supervised` mode should run:

1. worker round
2. technical validation
3. if validation fails, repair or fail without launching supervisor
4. if validation passes, launch supervisor audit
5. if supervisor reports `revise-metadata`, `revise-required`, or `hard-fail`, handle that result
6. if supervisor passes and `min_rounds` is not yet satisfied, continue with injected CLI plus supervisor summaries
7. otherwise stop and return the worker result

### Supervisor Decision Contract

Supervisor should no longer decide whether the loop stops.

Supervisor decisions should be limited to:

- `pass`
- `revise-metadata`
- `revise-required`
- `hard-fail`

The supervisor prompt should explicitly say that the CLI decides whether another round is required from the round-loop policy, and that the supervisor should not encode stop/continue in its decision line.

### Prompt Summary

The CLI summary passed into later rounds should separate:

- technical decision: `pass`, `revise-required`, or `hard-fail`
- whether another round is required
- issues or warnings from the technical check

That avoids using `pass-stop` and `pass-continue` as overloaded pseudo-decisions.

## Compatibility

For a short compatibility window, the runtime may treat legacy supervisor report values `pass-stop` and `pass-continue` as `pass` when reading existing report content generated by older prompts.
