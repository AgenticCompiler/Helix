# Backend Workspace Directory Cleanup Design

## Problem

After agent runs complete, backend-specific directories (`.opencode/`, `.codex/`, `.pi/`, `.claude/`, `.openhands/`, `.traecli/`) remain in the workspace. Currently, triton-agent only cleans up the files it explicitly staged:

| What | Cleaned by |
|------|------------|
| `.opencode/skills/*` | `SkillLinkManager.cleanup()` |
| `.opencode/plugins/triton-agent-hook-guard.js` | `AgentHookManager.cleanup()` |
| `.opencode/triton-agent-hooks/*` | `AgentHookManager.cleanup()` |
| `.opencode/node_modules/` | **NOT cleaned** |
| `.opencode/package.json` | **NOT cleaned** |
| `.opencode/package-lock.json` | **NOT cleaned** |

The last three are generated by the opencode CLI tool itself during initialization and are unknown to triton-agent. Same applies to other backends (codex, pi, claude, etc.) that may generate their own runtime artifacts under their backend directories.

## Design

### Approach

Extend `SkillLinkManager` to track and clean the entire backend root directory (e.g., `.opencode/`), not just the `skills/` subdirectory within it. Use an ownership check to avoid deleting user-pre-existing directories.

### Safety Mechanism: Ownership Tracking

Before staging any skills, `prepare_skills()` checks whether the backend root directory already exists. If it does NOT exist, triton-agent "owns" it and will clean it up entirely after the run. If it DOES already exist (user may have pre-created it with their own config), triton-agent leaves it alone.

This follows the AGENTS.md principle: "Never delete or replace user-owned files or directories during cleanup." Backend directories that triton-agent creates are not user-owned.

### Cleanup Order

The existing cleanup order is:

1. **Hooks cleanup** (inner `finally` in `AgentRunner.run()`) — removes `.opencode/plugins/` and `.opencode/triton-agent-hooks/`
2. **Skills cleanup** (outer `finally` in orchestration) — removes `.opencode/skills/`

The new backend root cleanup runs at step 2, after all skill dirs have been removed. By this point, hooks are already cleaned up, and the backend root only contains backend-generated artifacts (e.g., `node_modules/`, `package.json`). The entire backend root is then removed via `shutil.rmtree()`.

### Scope

Affects all six backends: `opencode`, `codex`, `pi`, `claude`, `openhands`, `traecli`.

### Changes

#### `src/triton_agent/skills.py`

1. Add `backend_root: str` field to `_SkillBackendConfig`:

```python
@dataclass(frozen=True)
class _SkillBackendConfig:
    target_parts: tuple[str, ...]
    copy_root_when_missing: bool
    backend_root: str  # e.g., ".opencode" — the top-level backend directory to clean
```

2. Update `_SKILL_BACKEND_CONFIGS` to include `backend_root` for each backend:

| Backend | `backend_root` |
|---------|---------------|
| codex | `.codex` |
| opencode | `.opencode` |
| pi | `.pi` |
| claude | `.claude` |
| openhands | `.openhands` |
| traecli | `.traecli` |

3. In `prepare_skills()`, before staging, check if the backend root exists. If not, prepend it to `created_paths` so it gets cleaned last (reversed iteration in `cleanup()`):

```python
backend_root_path = workdir / config.backend_root
root_pre_existed = backend_root_path.exists()

# ... existing staging logic ...

# If the backend root didn't exist before we staged, we own it and will clean it
if not root_pre_existed:
    created.insert(0, backend_root_path)
```

4. No changes to `cleanup()` — it already removes all `created_paths` via `shutil.rmtree()`.

#### Other files

No changes needed. All orchestration files (`convert/orchestration.py`, `generation/orchestration.py`, `optimize/orchestration.py`) already call `prepare_skills()` and `cleanup()` with the same pattern.

### Behavior Matrix

| Scenario | Backend root pre-existed | Staged skills | Cleanup result |
|----------|------------------------|---------------|----------------|
| Fresh workspace | No | Yes | Entire backend dir removed |
| Workspace with leftover backend dir from previous incomplete run | Yes | Yes | Only skills removed, backend dir (with old artifacts) left behind |
| Workspace with user's own `.opencode/` config | Yes | Yes | Only skills removed, user's `.opencode/` preserved |

### Edge Cases

- **Crash before cleanup**: Same as current behavior — staged files are left behind. No regression.
- **Hooks disabled (`--pure` mode)**: Skills are still staged and cleaned; backend root cleanup proceeds normally since hooks aren't involved.
- **`enable_agent_hooks=False`**: Same as above — hooks prepare/cleanup is skipped, but skills cleanup still runs and removes backend root.
- **Multiple skills prepare for same backend root**: `prepare_skills()` is called once per agent run. No conflict.

### What This Does NOT Address

- Cleaning up backend directories that pre-existed from previous incomplete runs (e.g., the first run left `node_modules/`, the second run finds `.opencode/` already exists and skips cleanup). This is acceptable because:
  - Workspaces are typically ephemeral
  - The leftover files are tool-generated, not user data
  - Users can manually `rm -rf .opencode` if needed
  - A future enhancement could add a CLI flag like `--clean-backend-dirs` to force cleanup
