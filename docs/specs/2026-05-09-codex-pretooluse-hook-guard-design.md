# Codex PreToolUse Hook Guard Design

## Goal

Add an opt-in Codex-first hook staging mechanism that blocks redundant agent reads of staged skill implementation files before the tool call executes.

The first policy should prevent Codex from reading files such as staged skill `scripts/` contents through shell commands. When blocked, Codex should receive a short actionable message that tells it the file does not need to be read and that it should rely on the skill instructions and documented command interface instead.

## User-Visible Behavior

- Agent hooks are disabled by default.
- The initial CLI surface is `optimize --enable-agent-hooks`.
- When `--enable-agent-hooks` is set for an optimize run, Codex-backed agent invocations should stage a temporary workspace-local hook before launching Codex.
- Non-Codex backends should ignore the hook request until backend-specific templates exist.
- The hook should intercept `PreToolUse` events for shell/Bash tool calls.
- If the shell command attempts to read protected staged skill implementation files, the hook should deny the tool call.
- The denial reason shown to the agent should be concise:

```text
This read is blocked by triton-agent workspace policy. Stay within the current workspace and do not inspect staged skill implementation files under .codex/skills/*/scripts/. Use the skill instructions and documented command interface instead.
```

- Non-matching commands should run unchanged.
- Existing user-owned `.codex` files should not be overwritten or removed.
- Cleanup should remove only hook files created by the current `triton-agent` run.

## Design

### Hook Templates

Store Codex hook assets in a repository-owned template directory:

```text
hooks/codex/
```

The directory should contain:

```text
hooks/codex/hooks.json
hooks/codex/pretooluse_guard.py
```

Keeping these files in `hooks/codex/` makes the hook behavior inspectable and avoids embedding large JSON or script bodies in Python string constants.

The staged workspace should receive copied hook assets under `.codex/`, using paths owned by the current run:

```text
.codex/hooks.json
.codex/triton-agent-hooks/pretooluse_guard.py
.codex/triton-agent-hooks/policy.json
```

The implementation should render `policy.json` during staging because the policy is workspace-specific. Runtime values should include the resolved workspace root and path rules derived from that root. The policy logic should live in the template script; the generated policy should provide data, not Python code. Unlike the plugin side, Codex does not need a template policy file in the repository.

### Staging Lifecycle

Add a hook staging manager parallel in spirit to `SkillLinkManager`. The shared runner should call this manager only when the `AgentRequest` explicitly enables agent hooks.

The manager should:

- Support backend-specific hook preparation, starting with `codex`.
- Copy or render the Codex hook templates into the target workspace before the runner launches.
- Track every path created by the current run.
- Refuse to overwrite an existing user-owned hook file or directory.
- Clean up tracked paths in reverse order after the agent exits.
- Leave any pre-existing `.codex` directory and user-owned contents untouched.

If `.codex/hooks.json` already exists, the first implementation should fail with a clear error rather than merge into it. Merging hook configuration safely can be added later, but fail-fast behavior is easier to reason about and avoids silently changing user hook semantics.

### Hook Configuration

The staged `hooks.json` should register a `PreToolUse` command hook for shell-like Codex tools. The matcher should cover the Codex tool name used for shell commands, with room to add aliases if Codex exposes them in future versions.

The hook command should invoke the staged guard script with the staged policy file:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "python3 .codex/triton-agent-hooks/pretooluse_guard.py --policy .codex/triton-agent-hooks/policy.json"
          }
        ]
      }
    ]
  }
}
```

Codex passes the active tool event to command hooks through stdin. The command line therefore only needs to identify the guard script and policy file; `pretooluse_guard.py` should read the `PreToolUseCommandInput` JSON from stdin and extract `tool_name` plus `tool_input` from that payload.

The exact JSON shape should match the Codex CLI hook format supported by the installed Codex version. Tests should validate the staged JSON content generated by the manager rather than requiring a live Codex invocation.

### Guard Policy

The generated policy should be tied to the target workspace. It should include:

```json
{
  "workspace_root": "/absolute/path/to/workspace",
  "allow_read_roots": ["/absolute/path/to/workspace"],
  "deny_read_globs": [
    "/absolute/path/to/workspace/.codex/skills/*/scripts/**"
  ],
  "deny_message": "This read is blocked by triton-agent workspace policy. Stay within the current workspace and do not inspect staged skill implementation files under .codex/skills/*/scripts/. Use the skill instructions and documented command interface instead."
}
```

The guard should deny read-oriented shell commands when they attempt to read either:

- paths outside `allow_read_roots`
- paths matching `deny_read_globs`, including `.codex/skills/*/scripts/**`

Path checks should normalize relative paths against the hook input `cwd` and then resolve them against `workspace_root`. The guard should reject paths that escape the workspace through `..` segments or absolute paths. Symlink behavior should be conservative: if a candidate path exists, resolve it before applying allow and deny checks.

The guard should inspect the `PreToolUse` input JSON from stdin and evaluate only shell/Bash tool inputs. It should detect common read-oriented commands that expose file contents, including:

- `cat`
- `sed`
- `head`
- `tail`
- `less`
- `more`
- `awk`
- `rg` commands that target protected implementation paths

The first implementation should be conservative. It should block when a read-oriented command clearly targets a forbidden path, but it should not attempt to parse every possible shell construct. It should prefer clear, low-risk matches over broad command rejection.

### Denial Output

When the guard blocks a command, it should write structured JSON to stdout:

```json
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "deny",
    "permissionDecisionReason": "This read is blocked by triton-agent workspace policy. Stay within the current workspace and do not inspect staged skill implementation files under .codex/skills/*/scripts/. Use the skill instructions and documented command interface instead."
  }
}
```

This uses Codex's hook-specific `PreToolUse` output shape so Codex receives the denial reason as tool feedback. The script should exit with status `0` after writing a valid denial JSON object.

If the command is allowed, the guard should emit no stdout and exit `0`.

If the hook input is malformed, the guard should fail open and optionally emit a diagnostic to stderr. Hook parser failures should not break otherwise valid agent runs.

### Current Codex Coverage Boundary

The first implementation should not claim complete file-read enforcement. Current Codex hook coverage is strongest for shell/Bash tool calls. The guard should therefore be documented and tested as shell-read protection, not universal read-tool protection.

If future Codex versions emit `PreToolUse` payloads for built-in file read tools, the same guard can be extended to inspect those tool names and inputs.

## Testing

Add focused unit tests rather than depending on a live Codex session.

- Add hook staging tests that verify Codex hook assets are created under `.codex/`, tracked for cleanup, and removed without deleting user-owned directories.
- Verify staging fails clearly when `.codex/hooks.json` already exists.
- Verify non-Codex backends do not receive Codex hook files.
- Add guard script tests for allowed in-workspace reads, blocked outside-workspace reads, blocked `.codex/skills/*/scripts/` reads, and malformed input fail-open behavior.
- Add CLI and optimize orchestration coverage showing `optimize --enable-agent-hooks` maps to the agent request, while default optimize requests leave hooks disabled.
- Add runner coverage showing enabled Codex requests prepare hooks before launch and clean them up afterward, while default requests skip hook staging.

## Scope Boundaries

- Do not implement hook support for `opencode`, `pi`, `claude`, `openhands`, or `traecli` in this first change.
- Do not expose the hook flag on `optimize-batch` or other subcommands in this first change.
- Do not modify global `~/.codex/hooks.json` or user-level Codex config.
- Do not merge with pre-existing workspace `.codex/hooks.json` yet.
- Do not move workflow logic from skills into the CLI. The hook should protect the skill boundary; it should not duplicate skill behavior.
- Do not treat this as a security sandbox. It is an agent-behavior guardrail for redundant reads.
